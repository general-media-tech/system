## أولاً — إصلاح نقطة البيع `index-final.html`

### الأخطاء التي كانت موجودة:

#### ❌ خطأ 1: `validatePhoneField` غير معرّفة
```
الخطأ: oninput="validatePhoneField(this)" → ReferenceError
```
**الإصلاح:** استُبدلت بـ `gmtValidatePhone()` مع تعريف كامل:
- تتحقق من صيغة `09XXXXXXXX` (10 أرقام تبدأ بـ 09)
- تُظهر/تُخفي رسالة الخطأ `#phone-err`
- تضيف/تزيل `has-error` و`has-ok` تلقائياً
- الحقل الفارغ لا يُعتبر خطأ (الهاتف اختياري للبيع النقدي)

#### ❌ خطأ 2: `validateCheckout` لا تُعيد `true`
```javascript
// الكود القديم — الزر لا يعمل أبداً:
onclick="if(validateCheckout()){goTo('v-print');renderPrint()}"
// validateCheckout كانت تنتهي بـ renderLiveA4() بدون return true
```
**الإصلاح:**
- الدالة تُعيد `false` عند كل خطأ
- تُعيد `true` صراحةً عند النجاح
- فُصلت منطق الـ render عن منطق التحقق
- أُزيل استدعاء `hideLoad()` الخاطئ داخل validateCheckout

#### ❌ خطأ 3: `has-error` لا تُزال عند التصحيح
**الإصلاح:** أُضيف `classList.remove('has-error')` في بداية validateCheckout وفي `pickCust()`

#### ❌ خطأ 4: رسالة خطأ الهاتف مرئية دائماً
```html
<!-- قديم: display block دائماً -->
<div class="field-err" id="phone-err">يجب أن يبدأ بـ 09...</div>
<!-- جديد: مخفية بـ default -->
<div id="phone-err" style="display:none;">يجب أن يبدأ بـ 09...</div>
```

## ثانياً — نظام إدارة نقاط البيع في الأدمن `admin-final.html`

### الميزة الجديدة: تاب "🏪 نقاط البيع"

**المبدأ الذهبي للاستقرار:**
- `key_name` (haleb, homs...) = **لا يتغير أبداً** — مرتبط بكل الفواتير والمخزون القديمة
- `display_name` = **يتغير بحرية** — يظهر في كل الشاشات والفواتير والتقارير

**الميزات:**
1. عرض كل نقاط البيع مع أيقوناتها وأكوادها الداخلية
2. تعديل الاسم العربي مع معاينة فورية قبل الحفظ
3. تغيير الأيقونة (إيموجي)
4. إخفاء/إظهار الفرع من جداول الجرد
5. إضافة نقطة بيع جديدة مع تحذير SQL
6. تحديث فوري للذاكرة المحلية بدون إعادة تحميل

**التأثير على باقي النظام:**
- الفواتير الجديدة تظهر الاسم الجديد فوراً (تقرأ `display_name` من `branches`)
- الفواتير القديمة تبقى بالاسم القديم (محفوظ في `branch_name` كنص ثابت)
- جداول الجرد والتقارير: الاسم الجديد فور الحفظ
- نقطة البيع POS: تحتاج تحديث صفحة بعد تغيير الاسم

## ثالثاً — نظام النقل الداخلي (الجرد والأدمن)

### `transfer_setup.sql` — قاعدة البيانات:
```sql
branch_transfers  -- رأس فاتورة النقل
transfer_items    -- بنود الفاتورة
branch_notifications -- إشعارات الفروع
```
**نقاط مهمة:**
- `deduct_branch_stock()` RPC يستخدم `FOR UPDATE` لمنع race condition
- `add_branch_stock()` RPC لإضافة الكميات عند الاستلام
- Cascade Delete: حذف الفاتورة يحذف بنودها تلقائياً
- `UNIQUE` على `transfer_number` لمنع التكرار

### في `index__17_.html`:
- زر "نقل" في الشريط العلوي مع badge إشعارات أحمر
- مودال فاتورة نقل كامل: بحث منتجات، جدول بنود، ملخص أرباح
- معاينة وطباعة الفاتورة مع خانات التوقيع
- مودال تأكيد الاستلام: اسم المستلم + صورة (كاميرا أو رفع)
- سجل كل فواتير النقل مع فلتر حسب الحالة
- خصم فوري من الكاش المحلي (GMTCache) + تحديث Supabase

### في `admin-final.html`:
- تاب "🚚 نقل البضاعة" مع إحصاء سريع
- عرض كل الفواتير مع إمكانية تأكيد الاستلام ورفع الصورة مباشرة

## خامساً — ربط الأوردر بفاتورة البيع (الأدمن)

**الزر 🔗 في كل بطاقة فاتورة:**
1. يطلب رقم ID الأوردر (أو order_number)
2. يتحقق: الأوردر غير مقفل وغير مرتبط بفاتورة أخرى
3. يُظهر confirm مع تفاصيل الأوردر (الزبون، المبلغ)
4. عند الموافقة:
   - يحوّل الأوردر لحالة `prepared`
   - يُقفل الأوردر (`is_locked: true`)
   - يُصفّر عمولة الفاتورة (فواتير الشحن بلا عمولة)
   - يسجّل `linked_invoice_id` و`prepared_by` و`prepared_at`

## سادساً — `gmt-tour.js` النظام التعليمي

### البنية:
```
GMTLoadingScreen  → شاشة تحميل أمامية مع شريط تقدم
gmtShowWelcome()  → 5 شاشات ترحيبية (تظهر مرة واحدة فقط)
GMTTour class     → جولة تفاعلية تُحاط العناصر
```

### خطوات الجولة:
- **POS:** 7 خطوات (بحث، سلة، بيانات العميل، حفظ...)
- **الجرد:** 8 خطوات (وضع الجرد، جماعي، مجموعات، نقل، إشعارات...)
- **الأدمن:** 6 خطوات (فواتير، ربط أوردر، نقل، عمولات، عروض...)
- **الأوردرات:** 6 خطوات (التبويبات، الفلتر...)
- **المشتريات:** 3 خطوات

### كيفية الاستخدام:
- يظهر تلقائياً للمستخدم الجديد
- زر "🎓 إعادة الجولة" متاح دائماً
- الحالة محفوظة في `localStorage` بـ key: `tour_done_{pageId}`

## ما يحتاج إجراء يدوي من Supabase

### 1. تشغيل SQL:
```sql
-- شغّل transfer_setup.sql كاملاً من SQL Editor
```

### 2. إنشاء Bucket يدوياً:
```
Storage → New Bucket
الاسم: transfer-signatures
النوع: Private
```

### 3. بعد إضافة فرع جديد من الأدمن:
```sql
ALTER TABLE products ADD COLUMN IF NOT EXISTS {key_name} INTEGER DEFAULT 0;
-- مثال:
ALTER TABLE products ADD COLUMN IF NOT EXISTS damascus2 INTEGER DEFAULT 0;
```

### 4. للتحقق من نجاح الـ SQL:
```sql
SELECT next_transfer_number();
-- النتيجة المتوقعة: TRF-2026-06-000001
```
